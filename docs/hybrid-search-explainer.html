From Vector Search → Hybrid Search

What changed in the LearnMate RAG retrieval pipeline, and why.

Dense / vector (meaning) Sparse / BM25 (exact words) RRF fusion (merge) Cross-encoder rerank (precision)

1. The one-line idea

Before: we found chunks by meaning only (vector search). If the right chunk happened to use different words than the question, the embedding model could rank it low.

After (hybrid): we also find chunks by exact words (keyword / BM25 search), then merge the two lists. Now a chunk wins if it matches the meaning or the literal terms — ideally both. A final reranker sorts the merged shortlist.

Why it matters here: your notes are full of exact tokens like to_thread, 384-dim, worker_threads. Meaning-search can blur those; keyword-search nails them.

3. After — hybrid flow

The single recall step splits into two parallel retrievers that are then fused. Everything downstream (rerank → LLM) is unchanged.

User question
Path A · meaning
Embed → Vector searchQdrant · returns ranked chunk ids
Path B · exact words
BM25 keyword searchover the session's chunks · ranked ids
Reciprocal Rank Fusion (RRF)merge the two id lists into one pool of ~20
Cross-encoder rerankreorder the fused pool · keep top 5
LLM answer
Key insight: BM25 needs the whole set of the session's chunks to score against (not just the vector hits) — otherwise it could never surface a chunk that vector search missed. That's exactly why we added get_session_chunks(): it loads the full corpus once, and that same load also acts as the "do we even have documents?" gate.

4. What each new piece does

NEW  app/services/keyword_service.py — BM25

A dependency-free keyword scorer. Given the question's words, it scores every chunk on three intuitions:

RulePlain meaningKnob
Term frequencyA chunk that uses the query word more is more relevant…k1 (saturation: extra repeats help less and less)
Inverse doc frequency (IDF)…but rare words (e.g. libuv) count far more than common ones (e.g. "the").—
Length normalizationA long chunk naturally contains more words, so we discount its matches.b (how hard to penalize length)

NEW  app/services/hybrid_service.py — RRF + orchestration

reciprocal_rank_fusion() merges ranked lists using only each chunk's position, never its raw score. hybrid_rank() ties it together: build BM25 over the session chunks, rank them, then fuse that with the dense ranking.

CHANGED  app/services/rag_service.py

Added get_session_chunks() → returns every {chunk_id, text} for the session. Rebuilds chunk_id the same way it's stored in Qdrant (document_id_index) so the two retrievers speak the same id language and can be fused.

CHANGED  app/services/claude_service.py

The retrieval block went from one step to the dense + BM25 + RRF sequence above. The gate switched from session_has_documents() to "is get_session_chunks() empty?" (one read does both jobs).

CHANGED  evals/run_eval.py

Now scores four configurations so we can measure each piece in isolation: vector only, hybrid only, vector+rerank, hybrid+rerank.

5. RRF, worked with real numbers

RRF score for a chunk = sum over each list of 1 / (k + rank), with k = 60 and rank starting at 1. A chunk near the top of either list scores well; near the top of both wins. Take this question:

Read it: c0 wins because it ranked high in both retrievers. c2 was only 3rd by meaning but 1st by keywords — fusion rescues it to 2nd overall. Notice RRF never compared the raw cosine (~0.5) against the raw BM25 score (could be 8.9) — those live on totally different scales. It only used ranks, which is what makes it safe to merge two completely different scoring systems.

6. The code path, condensed

# app/services/claude_service.py  (inside get_claude_response)

session_chunks = await get_session_chunks(email, session_id)   # full corpus + gate
if session_chunks:
    # Path A — dense (meaning). encode = CPU → thread; Qdrant = async I/O → await
    q_vec       = await asyncio.to_thread(generate_embeddings, user_message)
    dense_pts   = await search_embeddings(q_vec, email, session_id, top_k=20)
    dense_ids   = [p.payload["chunk_id"] for p in dense_pts]

    # Path B + merge — BM25 over the corpus, then RRF (CPU → thread)
    fused_ids   = await asyncio.to_thread(hybrid_rank, user_message,
                                          session_chunks, dense_ids, 20)

    # Precision — cross-encoder reorders the fused pool, keep best 5
    candidates  = [point(text_by_id[i]) for i in fused_ids]
    top5        = await asyncio.to_thread(rerank, user_message, candidates, 5)

    document_context = "\n\n".join(c.payload["text"] for c in top5)

Hybrid clearly improves recall (vector-only → hybrid-only: hit@3 .85 → 1.00, MRR .768 → .850). After reranking the two tie — but only because this tiny 13-chunk test corpus fits entirely inside the 20-candidate pool, so the reranker sees the same chunks either way. On a real corpus (pool ≪ total), hybrid's better recall would feed the reranker better candidates and carry through to the final answer.
