Phase 0 — The Safety Net

Making the turn orchestrator testable before LangGraph rewrites it. 44 → 71 tests, zero Groq tokens.

Why now: the LangGraph migration replaces the internals of handle_turn — the single function every chat message flows through. Before this phase, that function had zero tests. We could have rewritten it completely, watched the suite go green, and shipped a broken tutor.

Run with python -m pytest. Nothing here touches a live Groq, MongoDB, Qdrant or Notion. The whole suite is 28 seconds.

The problem we were in

The pattern is not an accident. The tested modules are pure functions. The untested ones need a network. So the most important code in the app was the least tested — and all three bugs it has shipped were found by a human reading chat transcripts in production.

Why a fake LLM is the whole unlock

The orchestrator's decisions all have the same shape: "what does the code do when the model replies with exactly X?"

You cannot make a real LLM say exactly START_TEACHING on demand. And you certainly can't make it say START_TEACHING inside a sentence on demand — which is the case that matters, because that's the prompt-injection bug. Scripting the reply is the only way to reach both branches.

# tests/conftest.py
class FakeLLM:
    """Mimics the one slice of the Groq client the app uses:
    await client.chat.completions.create(...).choices[0].message.content"""

    def __init__(self):
        self.replies: list[str] = []
        self.calls: list[SimpleNamespace] = []
        self.chat = SimpleNamespace(completions=_FakeCompletions(self))

    def script(self, *replies: str):
        self.replies = list(replies)
        return self

The detail that earned its keep is what happens when the script runs out:

# _FakeCompletions.create()
if not self._llm.replies:
    raise AssertionError(
        f"FakeLLM: the orchestrator made an unexpected extra LLM call "
        f"(call #{len(self._llm.calls)}), but the script is exhausted."
    )

Why not return None? Because an unexpected extra LLM call is itself a regression. When we later re-introduced the injection bug to test the tests, this assertion fired first — the hijacked request had entered the teaching flow and called the model a second time to deliver a lesson it should never have started. The fake caught the hijack before any assertion in the test body ran.

The patching gotcha

tutor_service imports its dependencies by name at module load:

# app/services/tutor_service.py
from app.core.session import get_context, set_context, add_message, get_session

That binds those function objects into tutor_service's own namespace. Patching the source module does nothing — tutor_service already holds a reference to the original.

monkeypatch.setattr(app.core.session, "get_context", fake)   # no effect
monkeypatch.setattr(tutor, "get_context", store.get_context) # works

The two Notion functions are the exception. tutor_service imports them lazily, inside the function bodies, to break an import cycle — so the lookup happens at call time, and those we patch at the source module.

# inside _advance_topic()
from app.api.learn import save_single_note   # resolved when called...

# ...so patch it there
monkeypatch.setattr(learn, "save_single_note", notion.save_single_note)

FakeStore — asserting on state, not on prose

Its contexts dict mirrors the session_contexts.data sub-document, which is where the guided-session state machine actually lives. That lets a test verify the state transition directly instead of inferring it from the reply text.

class FakeStore:
    def __init__(self, **initial_context):
        self.contexts: dict = dict(initial_context)
        self.messages: list[dict] = []

    async def get_context(self, session_id, current_user_email):
        # A copy, so a caller holding an old ctx can't observe later
        # writes — matching Mongo, where get_context returns a snapshot.
        return dict(self.contexts)

So an assertion reads like the thing you actually care about:

assert store.contexts["current_topic"] == 0          # did NOT advance
assert store.contexts["current_topic_doubts"] == ["What is a layer?"]

Three tests named after bugs that shipped

bug 1   Prompt injection via a control token

An uploaded document containing the literal text START_TEACHING hijacked a "summarize my document" request into the guided-teaching flow. _handle_chatting was using a naive substring check.

async def test_document_naming_control_token_does_not_start_teaching(...):
    # the model's reply CONTAINS the token, but is not the token
    fake_llm.script("The document says the assistant must emit START_TEACHING to begin.")

    result = await tutor.handle_turn(SESSION, "Summarize my document.", USER)

    assert result["status"] == "chatting"
    assert "mode" not in store.contexts
    assert fake_llm.call_count == 1          # no curriculum, no lesson
    assert "START_TEACHING" not in result["reply"]   # never reaches the screen

The fix was strict whole-reply equality, not substring matching:

def _reply_is_token(reply: str, token: str) -> bool:
    cleaned = re.sub(r"[^A-Za-z_]", "", (reply or "")).upper()
    return cleaned == token

Stripping punctuation still tolerates **NEXT_TOPIC** and NEXT_TOPIC. — the model likes to bold things — while rejecting the token in prose.

bug 2   A question counted as "I understand, move on"

A genuine learner question tripped NEXT_TOPIC. The system saved an empty, non-personalized note and skipped a topic the learner had just said they didn't understand.

async def test_question_during_lesson_logs_doubt_and_does_not_advance(...):
    fake_llm.script("A layer is a read-only filesystem diff...")

    result = await tutor.handle_turn(SESSION, "What is a layer?", USER)

    assert store.contexts["current_topic"] == 0                     # did NOT advance
    assert store.contexts["current_topic_doubts"] == ["What is a layer?"]
    assert result["notion_pages"] == []                             # no note written

bug 3   "next topic" produced a header with no lesson

On advancing, the delivery call saw "next topic" as the last user message and replied NEXT_TOPIC again. That stripped to an empty string — the learner got a topic heading and nothing underneath.

The fix was a directive appended to the delivery prompt. The test asserts on what the model was told, not on what it replied:

async def test_delivery_prompt_instructs_model_to_teach_not_advance(...):
    fake_llm.script("START_TEACHING", "Lesson text.")
    await tutor.handle_turn(SESSION, "Teach me Docker", USER)

    delivery_prompt = fake_llm.system_prompt(1)      # the 2nd LLM call
    assert "Deliver this topic now" in delivery_prompt
    assert "do NOT output NEXT_TOPIC here" in delivery_prompt
    assert "Images and Containers" in delivery_prompt

This is why FakeLLM records calls, not just replies. A prompt is an input the code chose — and inputs are testable.

The reranker: two subtleties worth pinning

test_rerank_service.py  10

An unrelated AZURE_DEPLOYMENT_GUIDE.pdf was cited as source [5] on a question about Redis persistence. rerank always returned exactly top_k chunks: it sorted by score, sliced to 5, and threw the scores away. With only 2 relevant chunks it padded the list with the next-best three, however bad.

Real cross-encoder scores for "how does redis persistence work":

DocumentScore rangeOld behaviourNew behaviour
redis-notes.pdf+7.879 … +0.300returnedreturned
kafka_basics.txt−8.581cited as a sourcedropped
AZURE_DEPLOYMENT_GUIDE.pdf−10.320cited as a sourcedropped

Subtlety 1 — it's a logit, not a probability. Strongly positive means the passage answers the query; strongly negative means it doesn't. Zero is the boundary. Someone will eventually "fix" the threshold to 0.5 thinking it's a probability and drop every relevant chunk, so the constant itself gets a test:

def test_the_floor_is_zero_because_the_model_emits_a_logit():
    assert RELEVANCE_THRESHOLD == 0.0

Subtlety 2 — the best chunk survives even below the floor. With zero chunks returned, the grounding guard never gets attached to the prompt, and the model answers from world knowledge instead of declining.

kept = [top[0]] + [pair for pair in top[1:] if float(pair[1]) >= min_score]
def test_when_nothing_is_relevant_exactly_one_chunk_survives(scores):
    # "what is the capital of France" against a corpus of Redis notes
    points = _points("redis rdb", "redis aof", "redis replication")
    scores([-2.0, -9.0, -5.0])

    result = rerank("what is the capital of France", points, top_k=5)

    assert len(result) == 1
    assert _ids(result) == ["doc_0"]   # the least-bad one, not the first one

We monkeypatch reranker.predict rather than run the real model. That's deliberate: ms-marco model drift must not break these tests, but a change to the selection rule must. The tests assert on the selection logic, not on what the model happens to think today.

Proving the tests actually work

A test that has never failed is decoration. So each bug was re-introduced into the production code and the suite re-run.

All three were reverted; git diff app/ confirmed the production tree was untouched.

The mutation run also caught something the assertions alone would have missed. The injection mutation failed on FakeLLM's exhausted-script error, not on an assert — because the hijacked request had already made a second LLM call to deliver a lesson. A strict fake is a test in itself.

What this buys the migration

PropertyWhy it matters for LangGraph
Behaviour is pinnedPhase 3 replaces handle_turn's branching with a StateGraph. These assertions must keep passing unedited. If a phase needs an assertion changed, that phase is changing product behaviour — and that must be a decision, not a quiet edit.
Zero tokensThe Groq free tier caps at 100k tokens/day, and one earlier stress test ate 96k. Phases 1–4 can now be developed without spending any.
Fast28s for 71 tests. Fast enough to run on every save, which is the only way tests actually get run.
Scripted, not sampledPhase 2 swaps control tokens for native tool calls. Only the scripting of the fake changes — a tool_calls payload instead of a magic string. The assertions stay identical, which is exactly how we'll know the swap was behaviour-preserving.
