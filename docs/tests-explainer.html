Test Suite

33 unit tests guarding the core RAG logic — added before the SSE refactor as a safety net.

Why now: the streaming (SSE) change rewrites get_claude_response into an async generator and reorders its post-processing — exactly the kind of refactor that breaks things silently. These tests characterize the current behaviour so any regression during that refactor lights up immediately.

Run with python -m pytest. All tests are synchronous and need no live Mongo / Qdrant / Groq — only the MiniLM tokenizer loads (once).

A test earned its keep on day one

The first version of test_guard_wraps_context_in_matching_fence assumed the fence token appeared twice. It actually appears four times — twice in the instruction prose ("the text between <ctx-…> and </ctx-…>") plus the two real fence tags. The failing test forced the assertion to check the real property: all tokens identical + body wrapped in a matching fence. That's tests catching a wrong assumption before it becomes a wrong belief.

Scope — and what's next

These are unit tests for pure logic (the RAG algorithms + guard/citation helpers). To make the citation logic testable, mark_cited_sources was extracted from get_claude_response (same behaviour, now isolated) — which also leaves that function cleaner for the SSE rewrite.

Not yet covered (a future second layer): integration / endpoint tests — auth flow, file upload, the /message request contract, and DB cleanup on reset. Those need mocked clients or a test database and are a bigger effort. The unit layer here is the one that actually protects the SSE refactor.
