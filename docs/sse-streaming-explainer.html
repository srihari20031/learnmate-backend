Server-Sent Events — How the Streaming Works

A code walkthrough of the /api/chat/message/stream endpoint.

1. The problem & the idea

Before: the browser sends a request and waits for the entire answer to finish generating (several seconds) before it sees a single word — one request, one big JSON response.

With SSE: the server keeps the HTTP connection open and pushes many small events down it as the model produces tokens. Same total time, but the user sees text appear as it's typed. SSE is just a long-lived HTTP response with a special content type (text/event-stream) and a simple text format: each event is data: {json} followed by a blank line.

2. The flow

1Retrieve (hybrid + rerank)runs once, up front — sources are known before any token
2Stream tokens from Groqasync generator yields "delta" events as the model types
3Tail-buffer filters the sentinelsuppresses READY_TO_GENERATE so it never leaks
4"done" eventfinal text, cited flags, notion_urls

3. The event format

Every event is one small helper away. Each is a JSON object on a data: line, ending in a blank line — and each carries a type the frontend switches on.

def _sse(payload: dict) -> str:
    # One Server-Sent Event: a JSON object on a `data:` line. Each event has a
    # "type" (sources | delta | status | done | error) the frontend switches on.
    return f"data: {json.dumps(payload)}\n\n"
EventPayloadFrontend does…
sources{sources: [...]}render citation chips (sent first)
delta{text: "..."}append to the live bubble
status{status: "generating_notes"}show a spinner
done{response, status, notion_urls, sources}finalize the message
error{message: "..."}show an error

4. The async generator (the heart of it)

A normal function returns once. An async generator yields many values over time, and awaits in between — perfect for "produce an event, wait for the next token, produce another." FastAPI streams each yielded string straight to the client.

async def get_claude_response_stream(session_id, user_message, current_user_email, attachments=None):
    # Same pipeline as the blocking version — but the LLM output streams.
    document_context, sources = await retrieve_context_and_sources(
        session_id, current_user_email, user_message)

    # (1) Sources are known before generation, so send them up front.
    yield _sse({"type": "sources", "sources": sources})

    await add_message(session_id, "user", user_message, current_user_email, attachments=attachments)
    history  = await get_session(session_id, current_user_email)
    messages = build_llm_messages(history, document_context)   # + grounding/injection guard

    # (2) stream=True -> Groq returns an async iterator of token chunks.
    stream = await client.chat.completions.create(model=MODEL, messages=messages, stream=True)

    hold, full_raw, pending = len(SENTINEL) - 1, "", ""

    async for chunk in stream:
        delta = chunk.choices[0].delta.content or ""
        if not delta: continue
        full_raw += delta
        pending  += delta

        # (3) tail-buffer: strip any full sentinel, hold back a short tail (see below)
        cleaned = re.sub(SENTINEL, "", pending, flags=re.IGNORECASE)
        if len(cleaned) > hold:
            emit, pending = cleaned[:-hold], cleaned[-hold:]
            if emit:
                yield _sse({"type": "delta", "text": emit})
        else:
            pending = cleaned

    # flush the held tail (any trailing sentinel removed)
    tail = re.sub(SENTINEL, "", pending, flags=re.IGNORECASE)
    if tail:
        yield _sse({"type": "delta", "text": tail})

    # (4) post-processing — mirrors the non-streaming path
    reply = full_raw
    mark_cited_sources(reply, sources)                # flag which sources got [n]
    display_reply = strip_sentinel(reply)
    await add_message(session_id, "assistant", display_reply, current_user_email)
    await _maybe_extract_context(session_id, history, current_user_email)

    if SENTINEL.upper() in reply.upper():          # model is ready -> build notes as a final phase
        yield _sse({"type": "status", "status": "generating_notes"})
        from app.api.learn import generate_learning_notes  # deferred: avoids circular import
        result = await generate_learning_notes(session_id, current_user_email)
        yield _sse({"type": "done", "response": "Your notes are ready in Notion!",
                    "status": result["status"], "notion_urls": result["notion_urls"], "sources": sources})
    else:
        yield _sse({"type": "done", "response": display_reply,
                    "status": "chatting", "notion_urls": [], "sources": sources})

5. The tail-buffer (the tricky part)

The intake prompt emits a control token, READY_TO_GENERATE, when it wants to build notes. We must never show it. But if we stream raw tokens, the user would watch it appear letter by letter. The fix: hold back the last few characters so a half-arrived sentinel is never flushed, and strip any completed one.

Why hold back len(SENTINEL) - 1 = 16 chars?

A complete sentinel is 17 chars. If we always keep the last 16 unflushed, then any sentinel that's still arriving (16 chars or fewer so far) sits safely in the buffer until the next token completes it — at which point re.sub deletes it. Text older than the last 16 chars can't be part of an unfinished sentinel, so it's safe to emit.

Worked example — sentinel split across tokens

# tokens arriving:  "Done. "  "READY"  "_TO_"  "GENERATE"  " bye"

pending="Done. "              cleaned="Done. "       len 6  <= 16  -> hold all
pending="Done. READY"         cleaned unchanged     len 11 <= 16  -> hold all
pending="Done. READY_TO_"      cleaned unchanged     len 15 <= 16  -> hold all
pending="Done. READY_TO_GENERATE"  re.sub removes it -> "Done. "  len 6 -> hold
pending="Done.  bye"          # after " bye" arrives  -> flush "Done.  bye"

# RESULT streamed to user:  "Done.  bye"   (sentinel never appeared)
# full_raw still holds it -> we detect "ready" and trigger note generation.
Two variables do two jobs: pending is the cleaned buffer we flush from (sentinel stripped); full_raw is the raw accumulation we scan at the end to decide whether notes generation should fire. Splitting them is what lets us hide the token from the user and still act on it.

6. The FastAPI endpoint

StreamingResponse takes any async generator and pipes each yielded string to the client with the text/event-stream content type. The wrapper catches mid-stream errors and turns them into a final error event instead of a dropped connection.

@router.post("/message/stream")
async def message_stream(
    message: str = Form(...),
    session_id: str = Header(...),
    current_user: TokenData = Depends(get_current_user),
    attachments: list[UploadFile] = File(default=[]),
):
    await get_chat(session_id, current_user.email)
    processed = await process_message_attachments(attachments, current_user.email, session_id)

    async def event_source():
        try:
            async for event in get_claude_response_stream(
                    session_id=session_id, user_message=message,
                    current_user_email=current_user.email,
                    attachments=processed or None):
                yield event
        except Exception as e:
            yield f"data: {json.dumps({'type':'error','message':str(e)})}\n\n"

    return StreamingResponse(
        event_source(),
        media_type="text/event-stream",
        headers={
            "Cache-Control": "no-cache",
            "Connection": "keep-alive",
            "X-Accel-Buffering": "no",   # stop nginx from buffering the stream
        },
    )
Frontend gotcha: the browser's native EventSource API cannot send an Authorization header, and our auth is a Bearer token. So the frontend must consume this with fetch() + response.body.getReader(), not EventSource.

7. Shared code (no duplication)

The streaming and non-streaming responders share one pipeline. During this change the common parts were extracted so both call the same functions:

The blocking get_claude_response keeps working and returns (reply, sources); the old /message endpoint is untouched. The stream is purely additive.
