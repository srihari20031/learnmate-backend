Citation Generation + Grounding

Making answers traceable to their source chunks — what we added and changed.

1. What & why

Citation generation = the answer comes back with its receipts: the specific source chunks it was based on, so the user can verify instead of trusting blindly.

Grounding = instructing the model to answer only from those sources (and admit when it can't), rather than from its own memory.

Together they turn "here's an answer" into "here's an answer, based on these passages from your notes" — the single biggest trust win for a learning tool, and a direct reducer of (perceived and real) hallucination.

2. The flow

Retrieve (hybrid + rerank)top-5 chunks, each still carrying its chunk_id
Number the context + resolve filenames[1] (from notes.pdf) <text> … one Mongo lookup for names
LLM answers with grounding instruction"use ONLY these sources; cite them inline with [n]"
Validate [n] markersflag which in-range sources were actually cited
Return answer + sources[]each with filename, snippet, cited flag
Key link: the [n] the model writes and the id in the sources array come from the same enumeration — the numbered context block and the source list are built in one loop, so [2] in the answer always maps to sources[id=2].

5. Design decisions worth remembering

DecisionWhy
Don't strip [n] from the answer text This is a coding tool — arr[2] / list[0] are legitimate. We only read markers to set the cited flag; we never mutate the text, so code is never corrupted.
Only in-range numbers count With 5 sources, [9] is ignored — it's either a hallucinated citation or code. Out-of-range never flips a cited flag.
Grounding scoped to "uploaded material" A hard global "answer only from context" would break the intake conversation (which is a general chat to build the curriculum). We scope it to questions about the documents.
Keep all sources, flag cited We return every retrieved source (not just cited ones) so nothing is hidden, but mark which the model used so the UI can emphasize them.
Backward compatible sources defaults to []; empty on plain-chat turns and on the notes-generation path. Old frontends ignoring it keep working.
Known limitation (false positive): if the answer contains real code like arr[2] and there is a source #2, that source gets flagged cited even if the code wasn't a citation. This is intentional — a harmless over-flag is far better than corrupting code by stripping brackets. If it ever matters, a distinctive citation token (e.g. [[2]]) would remove the ambiguity.
